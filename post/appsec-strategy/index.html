AppSec Strategy

Automated & Manual Efforts

 Posted on March 12, 2018  |  4 minutes (710 words)

Application security teams face a daunting challenge: finding and fixing security issues across their application portfolio, based on risk. Application portfolios tend to grow over time through acquisitions, partnerships, and in response to evolving business needs. New app development, COTS app deployment, and SaaS app/service adoption also expand the portfolio.

Manual security efforts such as penetration testing, code review, and design analysis are a great start, but scale poorly across large application portfolios. The industry lacks enough people with the right skills.

SAST, DAST, vulnerability scanners, and other tools can find some issues at greater scale than manual efforts. Tools alone aren’t enough – sophisticated attackers combine OSINT, active reconnaissance, and targeted attacks to exploit vulnerabilities that automation misses.

This post:

  • Explores ways to combine automated and manual discovery of security issues
  • Examines security issues best-suited to automated or manual discovery

Better Together

Automation offers shallower defect discovery at greater scale; manual efforts offer deeper defect discovery at lesser scale. They complement each other nicely.

Focus on high-risk applications first, such as:

  • Profitable line-of-business applications
  • Internet-facing applications handling sensitive customer data
  • Legacy applications developed without regard for security
  • Applications where a breach (or lateral movement post-breach) could cause serious financial, reputational, or compliance repurcussions

Automate discovery for the low-hanging fruit issues. This frees the application security team to focus on higher-value activities like:

  • Threat modeling
  • Security unit & integration testing
  • Goal-oriented red teaming
  • Security training
  • Security automation
    • Integrating application security tooling/testing with CI/CD² pipelines
    • Filtering tool-generated false positives
    • Expanding coverage to new applications

Creativity & Manual Efforts

Manual efforts are best for finding defects that require a semantic understanding of a system and creative thinking about how to break it. Tools generally lack context and understanding of an application or system, and thus miss certain classes of defect. These defects often come with substantial business risk.

Examples:

  • Business logic flaws: tools don’t understand the intended design/purpose of an application. Consider a concert or airline ticketing system with expiring ticket offers. Unpurchased tickets return from the shopping cart to the purchase pool after some time period. Could an attacker re-add the ticket to their cart every few minutes, blocking other customers from purchasing tickets? Could this be scripted to lock large numbers of tickets, resulting in lost sales?
  • Insecure direct object references: tools often won’t understand the meaning of parameters/fields in an application; they miss which fields are likely to represent files, database rows, or other directly-referenced objects. The Panera leak provides an example of this; Dylan Houlihan (a security researcher) found an unauthenticated API endpoint that exposed sensitive customer data via a sequentially-incrementing ID field.
  • Privilege escalation: tools rarely understand application privilege levels, how users map to these privilege levels, and how the privilege levels determine access to application functionality. For example, say a forum has a vulnerability allowing users delete each other’s posts. Normal users shouldn’t be able to do this, but administrators should. Tools generally lack this context around privilege levels, and miss privilege escalation defects.

Automate Low-Hanging-Fruit

Automation is best for uncovering well-defined, relatively static security issues. Examples:

  • Weak SSL/TLS configurations via tools like:
    • sslyze
    • sslscan
    • Qualys SSL Labs
  • Sensitive data exposure via tools like:
    • gitrob (scan GitHub organizations for sensitive info)
    • AWS Trusted Advisor S3 checks or s3-inspector (find public AWS S3 buckets)
    • git-secrets (prevent accidentally committing secrets to version control)
  • Use of known-vulnerable components via tools like:
    • Retire.js (flag outdated JavaScript libraries)
    • OWASP Dependency-Check (flag outdated Java and .NET dependencies)
    • Clair (flag vulnerable container components)

Automating static and dynamic analysis to find security defects in CI/CD² pipelines is a great way to scale defect discovery across applications. If you’re curious how this might look in practice for web applications, check out this lab that I built.

The lab is based on RailsGoat, an intentionally vulnerable web application from OWASP. It walks through discovering application security defects in RailsGoat with Brakeman (static analysis against the RailsGoat source) and Arachni (dynamic analysis against a running RailsGoat instance). The lab is orchestrated by Vagrant, Jenkins, Docker, and Docker Compose, modeling the use of common DevOps tooling and a build pipeline.

Wrap-Up

Automate discovery for the easier-to-find stuff, use skilled manual efforts for the harder-to-find stuff. Start with high-risk applications. Use threat modeling to guide decision-making.
